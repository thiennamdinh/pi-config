# Persistent Multi-Agent Planning

Goal: build local, persistent, named Pi agent workspaces that can be switched in-process, invoked non-interactively, and used for lightweight delegation, consultation, communication, and background/sidecar work.

## Core concept

An **agent** is a named local workspace containing exactly one persistent Pi session plus its own instructions, memory, skills, config, and message bus.

Agents are local-only and not committed to the Pi config repo, but each agent directory should be self-contained enough to tar up and move to another machine.

```text
~/.pi/agents/
  music/
    AGENTS.md
    manifest.json
    memory/
    skills/
    session/
    messages.jsonl
  scout/
    AGENTS.md
    manifest.json
    memory/
    skills/
    session/
    messages.jsonl
  reviewer/
    ...
```

Pi extensions and orchestration logic still live in the coordinating root, e.g. `~/.pi/agent/extensions/`. Agent workspaces contain agent-specific context and state, not extension source code.

## Principles

- Switching agents should happen **in-process** when possible, using Pi session/runtime switching rather than launching a separate pane.
- Treat `agent = one named session/workspace`; do not maintain multiple sessions per agent initially.
- Keep agents flat and open-ended: any number of named agents under `~/.pi/agents/<name>/`.
- Agents are local/private by default and excluded from git.
- Each agent owns an editable `AGENTS.md` for its local operating notes and self-maintained instructions.
- Each agent has separate durable memory and skills.
- Deterministic config lives in a manifest, not in hidden conversation state.
- A normal root Pi session is an ephemeral agent named `pi`.
- Ephemeral `pi` starts with no durable agent session and no agent-local context.
- Ephemeral `pi` can use `/agent-*` commands, but cannot be renamed.
- Ephemeral `pi` state may be discarded unless cloned into a persistent named agent.

## Agent workspace layout

```text
~/.pi/agents/<name>/
  AGENTS.md            # editable local agent instructions
  manifest.json        # deterministic model/tool/permission/session config
  memory/              # durable memory files for this agent
  .pi/skills/          # real Pi skills discovered because agent cwd is this home
  session/             # exactly one Pi session for this agent
  messages.jsonl       # canonical communication/inbox log
  artifacts/           # optional outputs generated by this agent or its clones
```

Agent cwd should be the agent home directory itself. This keeps each agent stable even when invoked from different terminals/projects with different current directories.

Possible manifest:

```json
{
  "name": "scout",
  "description": "Read-only codebase reconnaissance agent",
  "model": "openai-codex/gpt-5.5",
  "thinkingLevel": "medium",
  "tools": ["read", "grep", "find", "ls"],
  "permissions": {
    "edit": false,
    "bash": "read-only"
  },
  "cwdPolicy": "agent-home",
  "maxRuntimeSeconds": 300
}
```

## Actions / commands

### Agent lifecycle and switching

Pi extension commands are registered as flat slash commands, so `/agent-switch` style commands are likely more idiomatic and direct than a grouped `/agent switch` parser. Built-in Pi commands also commonly use hyphenated names, e.g. `/scoped-models`.

```text
/agent-list
/agent-new <name>
/agent-switch <name>
/agent-clone <new-name>
/agent-rename <new-name>
/agent-delete          # deletes current agent; requires confirmation
```

Desired behavior:

- `agent-new` creates a new persistent named agent workspace.
- `agent-switch` changes the current Pi runtime/session to the target agent in-process.
- `agent-clone` clones the current agent workspace/session to `<new-name>` and switches to it or offers to switch. Cloning ephemeral `pi` is how a normal root session becomes persistent.
- `agent-rename` renames the current persistent agent workspace/session to `<new-name>`. Renaming ephemeral `pi` is blocked.
- `agent-delete` deletes the current persistent agent workspace only, requires explicit confirmation, and moves it to `~/.pi/agents/.trash/<name>-<timestamp>/`.

### Communication: ephemeral clone consultation model

Prefer **consulting a forked ephemeral clone** of a target agent over trying to remote-control the live target session.

Core mental model:

```text
caller agent/user
  → ephemeral clone of target agent
  → answer/artifact returned to caller
  → inbox note logged for the real target agent to absorb later
```

This makes cross-agent work a delegated consultation rather than live IPC. The real target agent can later see: “A temporary clone of you handled this while you were away.”

Primary commands:

```text
/agent-consult <agent> <question>   # run ephemeral clone, return answer, log result
/agent-tell <agent> <message>       # append inbox note only; does not run target
/agent-inbox [agent]                # list inbox/consultation notes
/agent-absorb [message-id]          # mark/read/merge notes for current agent
```

Semantics:

- `agent-consult` is **pull**: ask a target agent’s clone for advice or analysis and return the answer to the caller immediately.
- `agent-tell` is **push**: record information/task for the real target agent’s future attention; it does not wake or control a live TUI.
- Consultation results are logged both for the caller and target, so there is an audit trail and the target can absorb useful state later.
- The target’s live persistent session is not mutated by the clone unless/until the real target absorbs the note.
- The clone should run in the target agent’s home directory, with target `AGENTS.md`, manifest, memory, and agent-local skills loaded.
- Default clone tool policy should be conservative: read-only/no-tools first, then explicit scoped tool permissions if useful.

Canonical communication remains JSONL for machine readability, with Markdown artifacts for human-readable long answers.

Example messages:

```jsonl
{"id":"msg_1","type":"tell","from":"bartimaeus","to":"jane","body":"User wants media-hub sources reviewed later.","createdAt":"2026-06-03T20:00:00Z","status":"queued"}
{"id":"msg_2","type":"consult_request","from":"bartimaeus","to":"jane","body":"What media-hub source quality checks matter next?","createdAt":"2026-06-03T20:01:00Z","status":"done","artifactPath":"artifacts/consultations/2026-06-03T20-01-00Z_msg_2.md"}
{"id":"msg_3","type":"consult_answer","from":"jane:clone","to":"bartimaeus","replyTo":"msg_2","body":"Prioritize duplicate-rate, source freshness, and ideological diversity checks.","createdAt":"2026-06-03T20:02:00Z","artifactPath":"artifacts/consultations/2026-06-03T20-01-00Z_msg_2.md"}
```

Potential message fields:

```json
{
  "id": "msg_<id>",
  "type": "tell|consult_request|consult_answer|event",
  "from": "authoritative-agent-or-user-id",
  "to": "authoritative-agent-name",
  "replyTo": "optional-message-id",
  "body": "message text or concise preview",
  "artifactPath": "optional markdown artifact path",
  "createdAt": "iso timestamp",
  "status": "queued|processing|done|failed|absorbed",
  "readAt": "optional iso timestamp",
  "absorbedAt": "optional iso timestamp"
}
```

### User out-of-band tasking

Separate from agent-to-agent communication, support direct user tasking of another agent without requiring the current agent/session to know about it.

Status: run-now live cross-agent communication is no longer the first target. Prefer ephemeral clone consultations for immediate work and inbox notes for future work.

Use explicit commands rather than configurable dispatch behavior:

```text
/agent-task <agent> <message>
/agent-consult <agent> <message>
```

Examples:

```text
/agent-task environment-keeper "look into session switch ergonomics later"
/agent-task inquirer "research bounded multi-agent orchestration patterns"
/agent-consult jane "review media-hub source quality priorities"
```

Semantics:

- `agent-task` is user-originated enqueue-only work. It does not wake/run the target agent.
- `agent-consult` is user-originated run-now consultation. It spawns an ephemeral target-agent clone, blocks until the clone answers, returns the answer to the current user/session, and logs the result for the real target agent.
- `agent-consult` must not open or mutate the target agent’s persistent session by default. It can use a temporary session or `--no-session` plus explicit artifact logging.
- If tool use is enabled for a consultation, permissions should come from the target manifest plus a conservative consultation override.
- Neither command needs to enter or clutter the target agent’s live conversation.
- Useful for advisory agents like `jane`, `hermione`, or future `murderbot`; controller/effect agents may still need direct tools or later IPC.

## Ephemeral clone consultation technical architecture

The default cross-agent execution path should be a non-interactive ephemeral clone of the target agent.

### Consultation flow

```text
caller session
  ├─ validates target agent exists
  ├─ writes consult_request to caller and target logs
  ├─ creates temp workspace/session under target artifacts/tmp or agent tmp
  ├─ invokes Pi non-interactively as target clone
  ├─ captures assistant result, usage, status, and artifacts
  ├─ writes consult_answer and Markdown artifact
  └─ returns answer to caller
```

### Invocation shape

Possible first implementation:

```sh
PI_AGENT_EPHEMERAL_CLONE=1 \
pi -p \
  --session-dir ~/.pi/agents/<target>/artifacts/consultations/tmp/<id>/session \
  --no-context-files \
  --model <target-manifest-model> \
  --thinking <target-manifest-thinking> \
  --tools <consultation-tool-allowlist> \
  --append-system-prompt ~/.pi/agents/<target>/AGENTS.md \
  --append-system-prompt <generated target memory/inbox prompt> \
  "<wrapped consultation prompt>"
```

Alternative: use Pi SDK/RPC directly from the extension if that becomes cleaner. CLI subprocess is simpler and isolates crashes, stale contexts, and target-session ownership.

### Prompt wrapper

The generated clone prompt should include authoritative metadata and clear boundaries:

```text
You are an ephemeral consultation clone of agent <target>.
Your answer will be returned to <caller> and logged for the real <target>.
Do not assume your response is automatically absorbed by the live target session.
If you make durable observations the real target should remember, put them under "Notes for <target>".

Caller request:
<question>

Relevant caller-provided context:
<context>
```

The caller agent should choose what task/context to include. The wrapper should enforce identity, audit metadata, scope, and output expectations.

### State and logs

Store communication in JSONL plus human artifacts:

```text
~/.pi/agents/<caller>/messages.jsonl
~/.pi/agents/<target>/messages.jsonl
~/.pi/agents/<target>/artifacts/consultations/<timestamp>_<id>.md
~/.pi/agents/<target>/artifacts/consultations/tmp/<id>/session/...
```

Message log rules:

- caller log records request and answer preview/path;
- target log records request, answer preview/path, clone model, status, and whether absorbed;
- artifact stores full prompt metadata, answer, usage/cost estimate, and any stdout/stderr diagnostics;
- temp clone sessions may be retained briefly for debugging or pruned after successful artifact creation.

### Absorption model

The real target agent should not silently mutate just because a clone answered. Instead:

- `session_start` or `turn_start` injection can show unread inbox/consultation notes for the current agent.
- `/agent-inbox` lists unread notes.
- `/agent-absorb <id>` marks a note absorbed and can optionally append a concise custom message or memory note to the target’s persistent session/memory.
- The target can disagree with or supersede a clone result; clone output is advisory, not canonical.

### Tool/permission model

Start conservative:

- v1 consultations: no write tools; allow read/search tools only when explicitly requested or enabled in target manifest.
- Cyber/security agents default read-only sentinel mode.
- Editing, playback, network effects, or destructive operations require an explicit mode flag and should usually remain live-agent/user-mediated rather than clone-mediated.

This avoids the worst divergence: a clone changed the world, but the real target agent never lived through the change.

### Failure semantics

- If the clone fails, write a `failed` message and artifact with stderr/exit status.
- Return failure to caller without mutating target session.
- If target is live elsewhere, it does not matter; consultation uses a separate temp session.
- If rate-limited, caller can use `/prompt-later` or retry later; the target inbox still records failed attempts if useful.

## Live-agent IPC design (optional later layer)

Live IPC is no longer the default design. It is only needed for cases where the real live agent must act in its own TUI/session immediately, such as controller agents with side effects or workflows where current live context is essential.

Problem: Pi does not expose a built-in way for one process to poke an already-running interactive Pi session and trigger a new prompt loop. Extensions can call `pi.sendUserMessage()` inside their own process, and RPC mode can control a headless process it owns, but there is no native `pi --send-to-session <id>` equivalent.

If implemented later:

- When a persistent agent starts interactively, it creates a local Unix socket, e.g. `~/.pi/agents/<name>/ipc.sock`.
- The agent lock/owner file records `pid`, `sessionFile`, `cwd`, and `socketPath`.
- Other Pi processes inspect the lock before live run-now operations.
- Requests include ids and desired delivery mode.
- The owner serializes remote requests and returns final status plus final assistant text.
- Do not implement ambiguous fire-and-forget delivery; live IPC should be all-or-nothing with correlated responses.

Until this exists, use ephemeral clone consultation plus inbox logs for cross-agent reasoning, and direct local tools for simple side effects.

## Built-in session command guardrails

Pi's built-in session commands remain visible and cannot be cleanly removed or overridden through normal extension APIs. Built-ins are handled before extension command routing, so registering an extension command with the same name is not a reliable override.

Policy:

- Leave harmless/useful built-ins alone:
  - `/session` for inspecting the underlying Pi session
  - `/tree` for rollback/audit branch navigation
  - `/compact` for context management
- Prefer `/agent-*` wrappers for agent workflow:
  - `/agent-rename` instead of `/name`
  - `/agent-new` instead of `/new`
  - `/agent-switch` instead of `/resume`
  - `/agent-clone` instead of `/clone` or `/fork`

Guardrail implementation:

- Use lifecycle hooks where available to cancel/warn when built-ins would violate agent invariants.
- `session_before_switch` can guard `/new` and `/resume`-style switches.
- `session_before_fork` can guard `/fork` and `/clone`.
- For commands without cancellable hooks, document the preferred `/agent-*` wrapper and repair state on `session_start` only if safe.
- Do not rely on local patches or unsupported command overrides.

## Non-interactive invocation

There should be a way to target an agent from scripts, cron jobs, or systemd user timers.

Possible wrapper shape:

```sh
pi-agent <name> "do X"
pi-agent <name> -p "summarize queued messages"
pi-agent --agent scout -p "inspect this repo"
```

Native Pi shape would be ideal if extension/CLI support allows it:

```sh
pi --agent scout -p "inspect this repo"
```

Non-interactive agent invocation should:

- load the target agent manifest
- use its dedicated session
- include its `AGENTS.md`, memory, and skills
- apply its deterministic model/tool/permission config
- process queued `messages.jsonl` items when appropriate

## Message/session retention

Pi session files are append-only JSONL trees. Compaction can reduce what is sent to the model by adding summary entries, but it does not necessarily shrink/delete the underlying audit history. That model is acceptable for agent sessions.

Agent `messages.jsonl` should follow the same bias: retain by default for auditability, and add archival/compaction only if logs become noisy or large.

## Rollback / audit history

Use Pi's existing session tree behavior. No destructive pruning is required.

Desired behavior:

- Allow rollback to an older point in the agent session by moving the active leaf.
- Leave future branches/orphaned alternatives in the session for auditability.
- Continuing from the rolled-back point creates a new branch, matching Pi's existing model.

This is acceptable even if the active experience is not perfectly linear, because retained branches are useful audit history.

## Initial agent candidates

### music-controller / erik

- Spotify playback/control and music memory.
- Simple playback actions may not need agent consultation; direct async Spotify tools may suffice.
- If the real music agent needs to reason about recommendations or taste, `/agent-consult erik ...` is appropriate.
- Live IPC may still be useful later for controller-style side effects, but it is not the default cross-agent pattern.

### environment-keeper

- Maintains local Pi/editor/tmux/desktop configuration.
- Knows Pi docs paths and local config locations.
- Good target for deferring environment improvements and config cleanup.

### sync-web-developer

- Dedicated development agent for sync-web-related projects.
- Keeps project-specific context, implementation notes, and skills separate from the general assistant.

### jane / hermione / murderbot-style specialists

- Research/question-answering, media/network intelligence, writing/research precision, and security audit specialists.
- Good targets for `/agent-consult <agent> ...` because their main value is advisory analysis that can be returned as an artifact.
- Can specialize in web/docs/current-info investigation, code/security review, or writing critique and return compressed findings.

## MVP sequence

1. Add `~/.pi/agents/` to `.gitignore`.
2. Define the agent workspace layout and manifest schema.
3. Build `/agent-list`, `/agent-new <name>`, and `/agent-switch <name>` using in-process session switching.
4. Add clone/rename/delete lifecycle commands for the current agent, including special handling for ephemeral `pi`.
5. Add JSONL communication with `/agent-task` and `/agent-tell` as enqueue/inbox-only operations.
6. Add `/agent-consult` using ephemeral target-agent clones, Markdown artifacts, and caller/target message-log entries.
7. Add `/agent-inbox` and `/agent-absorb` so real target agents can review and incorporate clone work.
8. Add non-interactive wrapper support for targeting an agent.
9. Add rollback/audit-history navigation using Pi's existing session tree behavior.
10. Consider live-agent IPC only for controller/effect agents after consultation workflows prove useful.
